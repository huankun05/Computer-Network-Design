# 三次握手（TCP 连接建立）

## 概述
三次握手（Three-Way Handshake）是 TCP 协议建立连接的过程，通过三次报文交换，确保通信双方都准备好发送和接收数据。

## 为什么需要三次握手？

**要确认双方的收发能力都正常：**

| 需要验证 | 通过 |
|---------|------|
| 客户端发送能力 | 服务器收到 SYN |
| 服务器接收能力 | 服务器收到 SYN |
| 服务器发送能力 | 客户端收到 SYN+ACK |
| 客户端接收能力 | 客户端收到 SYN+ACK |

## 详解过程

### 第一次握手（SYN）
```
客户端 → 服务器: SYN (seq=x)
```
- 客户端发送 SYN 报文，不携带数据
- 初始序列号 seq=x（随机生成）
- SYN=1 标志位置位
- 客户端进入 **SYN-SENT** 状态

### 第二次握手（SYN+ACK）
```
服务器 → 客户端: SYN+ACK (seq=y, ack=x+1)
```
- 服务器发送 SYN+ACK 报文
- 确认号 ack=x+1（期望收到 x+1）
- 自己的序列号 seq=y（随机生成）
- 服务器进入 **SYN-RCVD** 状态
- 服务器为该连接分配资源

### 第三次握手（ACK）
```
客户端 → 服务器: ACK (seq=x+1, ack=y+1)
```
- 客户端发送确认报文
- 确认号 ack=y+1
- 报文可以携带数据
- 双方进入 **ESTABLISHED** 状态

## 状态图
```
客户端                      服务器
CLOSED                      LISTEN
  |                           |
  | SYN-SENT                  |
  |──SYN(seq=x)────────────→|
  |                           | SYN-RCVD
  |                          |
  |←──SYN+ACK(seq=y,ack=x+1)─|
  |                           |
  | ESTABLISHED               |
  |──ACK(seq=x+1,ack=y+1)──→|
  |                           | ESTABLISHED
```

## 为什么不是两次握手？
如果只有两次握手：
1. 客户端发送 SYN → 服务器
2. 服务器回复 SYN+ACK，服务器认为连接已建立

**问题**：如果第一步的 SYN 在网络中滞留了很久才到达服务器，服务器会认为这是新的连接请求，回复 SYN+ACK 并进入 ESTABLISHED 状态。但客户端可能已经关闭了！服务器会一直等待永远不会到来的数据，造成资源浪费。

**三次握手解决**：客户端必须回复 ACK 确认，服务器才会进入 ESTABLISHED 状态。

## 为什么不是四次握手？
- 三次握手已经足够验证双向通信能力
- 服务器可以将 SYN 和 ACK 合并成一个报文（SYN+ACK）
- 四次握手会造成不必要的延迟

## SYN 泛洪攻击（SYN Flood）
攻击者发送大量 SYN 但不完成握手，服务器为每个半开连接分配资源，耗尽服务器资源。

**防御手段**：
- SYN Cookie：不立即分配资源，通过 Cookie 验证后再建立连接
- 缩短 SYN 超时时间
- 增加半连接队列大小
