# 四次挥手（TCP 连接释放）

## 概述
四次挥手（Four-Way Handshake）是 TCP 协议释放连接的过程。由于 TCP 是全双工的，每个方向都需要独立关闭，所以需要四次交互。

## 为什么需要四次挥手？

TCP 是全双工通信，意味着客户端和服务器都可以同时发送数据。释放连接需要：
- 关闭客户端→服务器的方向
- 关闭服务器→客户端的方向

**如果只有三次**：被动关闭方可能在收到 FIN 后还有数据未发送完，不能立即关闭自己的方向。

## 详解过程

### 第一次挥手（FIN）
```
主动关闭方 → 被动关闭方: FIN (seq=u)
```
- 主动方发送 FIN 报文
- 表示"我没有数据要发送了"
- 进入 **FIN-WAIT-1** 状态

### 第二次挥手（ACK）
```
被动关闭方 → 主动关闭方: ACK (seq=v, ack=u+1)
```
- 被动方收到 FIN，回复 ACK
- 进入 **CLOSE-WAIT** 状态
- 主动方收到 ACK，进入 **FIN-WAIT-2** 状态
- 此时**被动方向仍然可以发送数据**

### 第三次挥手（FIN）
```
被动关闭方 → 主动关闭方: FIN (seq=w, ack=u+1)
```
- 被动方发送完剩余数据后，发送 FIN
- 表示"我也没有数据要发送了"
- 进入 **LAST-ACK** 状态

### 第四次挥手（ACK）
```
主动关闭方 → 被动关闭方: ACK (seq=u+1, ack=w+1)
```
- 主动方回复最后的 ACK
- 进入 **TIME-WAIT** 状态，等待 2MSL
- 被动方收到 ACK，进入 **CLOSED** 状态

## 状态图
```
主动关闭方                  被动关闭方
ESTABLISHED                ESTABLISHED
  |                           |
  | FIN-WAIT-1                |
  |──FIN(seq=u)───────────→ |
  |                           |
  | FIN-WAIT-2               | CLOSE-WAIT
  |←──ACK(ack=u+1)───────── |
  |   (可能继续收数据)        | (可能继续发数据)
  |                           |
  |                          | LAST-ACK
  |←──FIN(seq=w)─────────── |
  |                           |
  | TIME-WAIT                 |
  |──ACK(ack=w+1)─────────→ |
  | (等待 2MSL)               | CLOSED
  | CLOSED                   |
```

## TIME-WAIT 状态

### 为什么需要 TIME-WAIT？
主动关闭方在发送最后一个 ACK 后，进入 TIME-WAIT 状态，等待 **2MSL**（Maximum Segment Lifetime，报文最大生存时间）才会进入 CLOSED。

**原因 1：确保最后一个 ACK 被对方收到**
- 如果最后一个 ACK 丢失，被动方会重发 FIN
- 主动方在 TIME-WAIT 期间可以收到重发的 FIN 并再次回复 ACK

**原因 2：让旧连接的报文在网络中消亡**
- 防止旧连接的报文被误认成新连接的报文
- 2MSL 保证旧连接的报文全部在网络中消失

### 2MSL 的时长
| 系统 | 默认 TIME-WAIT |
|------|---------------|
| Linux | 60 秒 |
| Windows | 120 秒（可调） |
| MSL 典型值 | 30-60 秒 |

## 状态转换表
| 步骤 | 主动关闭方状态 | 被动关闭方状态 |
|------|---------------|---------------|
| 初始 | ESTABLISHED | ESTABLISHED |
| 第 1 次 | FIN-WAIT-1 | ESTABLISHED |
| 第 2 次 | FIN-WAIT-2 | CLOSE-WAIT |
| 第 3 次 | FIN-WAIT-2 → TIME-WAIT | LAST-ACK |
| 第 4 次 | TIME-WAIT | LAST-ACK → CLOSED |
| 超时后 | CLOSED | CLOSED |

## 同时关闭
双方同时发送 FIN 的情况也是可能的，此时双方都经历 FIN-WAIT-1 → CLOSING → TIME-WAIT → CLOSED。
